# إرشادات التحصين الأمني (Security Hardening)

يوثق هذا الملف الإجراءات الأمنية المتبعة في نظام **Robust Penny Scalper v7.0** لتأمين لوحة التحكم (Dashboard) وضمان تشغيل النظام بصلاحيات آمنة.

## 1. مصادقة لوحة التحكم (Dashboard Authentication)
لحماية البيانات الحساسة (مثل الأرباح/الخسائر، حالة الحساب، والعمليات)، تم تنفيذ نظام مصادقة يعتمد على رمز الوصول (Token-Based Authentication).

### كيفية العمل:
- يعتمد النظام على وجود إعداد `auth_token` ضمن قسم `[dashboard]` في ملف `config.toml`.
- **الرفض التلقائي (Fail-Safe):** إذا كان هذا الرمز فارغاً أو مفقوداً، سيرفض خادم `rpsd` بدء التشغيل تماماً.
- لحماية قنوات الاتصال، تتطلب جميع المسارات (سواء الـ HTTP Endpoint `/` أو مسار WebSocket `/ws`) تقديم الرمز الصحيح كمعامل استعلام (Query Parameter `?token=...`) أو كترويسة `Authorization: Bearer ...`.
- إذا لم يتم توفير الرمز أو كان غير صحيح، سيتم رفض الاتصال تلقائياً برمز حالة `401 Unauthorized`.

## 2. التشغيل بدون صلاحيات الجذر (Rootless Deployment)
تم التخلص من الحاجة لتشغيل النظام كمدير نظام (Root/Administrator). في السابق، كان النظام يتطلب كتابة ملفات UDS (Unix Domain Sockets) وحفظ الحالة داخل مسار `/var/run/rps` المحمي.

### إجراء النشر (Deployment Procedure):
- تم تغيير المسار الافتراضي للـ Sockets ليكون `/tmp/rps` (والذي يكون قابلاً للكتابة بواسطة المستخدم العادي).
- يمكنك تعديل مسار `runtime_socket_dir` في قسم `[system]` بملف `config.toml` إلى أي مسار مملوك للمستخدم الذي يُشغل النظام (مثل `~/.local/state/rps`).
- **التوصية:** قم بإنشاء مستخدم مخصص (Service User) باسم `rps_runner` بصلاحيات محدودة، وقم بتشغيل محرك الـ Rust وجسر الـ Python تحته لتطبيق مبدأ أقل الصلاحيات (Principle of Least Privilege).

## 3. القيود الأمنية المتبقية (Security Limitations)
بالرغم من تطبيق التحصينات أعلاه، لا يزال هناك بعض القيود الأمنية التي يجب مراعاتها في بيئة الإنتاج (Production):

1. **الافتقار إلى تشفير TLS/HTTPS المدمج:**
   - خادم لوحة التحكم يعمل عبر بروتوكولات HTTP/WS غير المشفرة (Plaintext).
   - إذا تم تعريض لوحة التحكم لشبكة خارجية باستخدام إعداد `allow_insecure_remote = true`، فإن الـ Auth Token والبيانات يتم نقلها كنص صريح مما يعرضها للاعتراض (Sniffing).
   - **الحل:** يجب وضع الخادم خلف وكيل عكسي (Reverse Proxy) مثل Nginx، Caddy، أو Traefik لإدارة وتأمين حركة المرور بشهادات SSL/TLS وتحويل الاتصالات إلى HTTPS/WSS.

2. **أمان نظام الملفات (File System Permissions):**
   - تأكد من أن مسار الـ `runtime_socket_dir` ومسار `configs/` يحملان صلاحيات `0700` و `0600` على الترتيب (حيث لا يمكن لأي مستخدم آخر على نفس الخادم قراءة `config.toml` واستخراج الـ Auth Token).
