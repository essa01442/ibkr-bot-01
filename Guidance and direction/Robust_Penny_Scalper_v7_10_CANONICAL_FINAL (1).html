2.3 TapeScore Formula

TapeScore = wR×ScoreR + wA×ScoreA + wLP×ScoreLP + wSpr×ScoreSpread
              + wAbs×ScoreAbsorption + wBLS×ScoreBidLift

wR   = 0.30   # Aggressive Ratio     (الأهم — ضغط الشراء)
wA   = 0.22   # Acceleration         (تسارع الصفقات)
wLP  = 0.22   # Large Print          (طباعة كبيرة)
wSpr = 0.13   # Spread Score         (كلما ضيق كلما أعلى)
wAbs = 0.08   # Absorption           (امتصاص العرض)
wBLS = 0.05   # Bid Lifting Sequence (رفع متدرج للـ Bid)
# ∑ weights = 1.00

عتبات الدخول:
  FULL_ACTIVE  + Normal:            TapeScore ≥ 72
  FULL_ACTIVE  + بعد هدف يومي:     TapeScore ≥ 82
  WARM_ACTIVE  + Surge Override:    TapeScore ≥ 67

4) مواصفات Python Bridge

الوظيفة الوحيدة: Receive → Encode → Send. لا شيء آخر.

4.1 ممنوع داخل Python (مطلق)

  • أي Filtering أو Watchlist أو Guards أو Scoring.
  • أي حساب R/A/LP أو Aggregation.
  • Heavy logging أو JSON encoding في hot path.
  • Blocking wait على socket داخل مسار الاستقبال.

4.2 Batching (إلزامي)

  • دفعة كل 5–10ms.
  • حد أقصى للدفعة: 512 حدث.
  • عند اكتمال الدفعة قبل الوقت: إرسال فوري دون انتظار المؤقت.

4.3 Backpressure (إلزامي)

  • إذا امتلأ socket buffer: لا تنتظر — أسقط وفق أولويات QoS.
  • ضغط مستمر > 1s: أرسل DataQuality = degraded فوراً → Monitor Only.

4.4 Encoding

  • MessagePack — للبداية السريعة وسهولة التصحيح.
  • FlatBuffers — للأداء الأعلى مع ثبات schema في الإنتاج.
  • Buffer reuse لتقليل allocations — لا new() في hot path.

4.5 Heartbeat

  • كل 250ms.
  • غياب ≥ 1 ثانية = DataQuality = degraded → Monitor Only + تنبيه.

5) المعمارية والتزامن (Rust / Tokio)

5.0 خريطة المهام (Tasks)

  1. BridgeRxTask — قراءة من UDS وفك ترميز الدفعات. منفصلة تماماً عن DataTask.
  2. DataTask — Router يوزع الأحداث على القنوات. bounded queue.
  3. SlowLoopTask — بناء Watchlist + pacing + ترقية/تنزيل Tier + حساب MTF/Correlation/Baseline.
  4. FastLoopTask — Tape + Guards + Trigger. O(1). لا I/O. لا allocations. سلطة قرار الدخول فقط.
  5. OMSTask — تنفيذ الأوامر + fills + timeouts + idempotency.
  6. RiskTask — Kill Switches + Loss Ladder. سلطة Monitor Only تعلو على كل طبقة.
  7. MetricsTask — Latency + Logs + Alerts. لا تتداخل مع FastLoop أبداً.
مبدأ تقسيم العمل: SlowLoop يحسب ويخزن (MTF، Correlation، L2 Baseline، Blocklist cleanup). FastLoop يقرأ فقط ما خزّنه SlowLoop.

5.1 Event Bus + Shared State

  • Event Bus: tokio::mpsc لكل قناة (bounded).
  • Watchlist snapshot: Arc<ArcSwap<WatchlistSnapshot>> — FastLoop يقرأ فقط، لا lock.

5.2 DataTask Queue Policy

  • Queue داخلية محدودة السعة (bounded).
  • عند الامتلاء: إسقاط حسب QoS (§3.1).
  • إذا فُقدت 3 دفعات متتالية → DataQuality = degraded + Monitor Only.

5.3 قواعد أداء حتمية للـ FastLoop

  • لا I/O من أي نوع بعد الإقلاع.
  • لا allocations (no Box, Vec::new, String::from داخل الحلقة).
  • Ring Buffers ثابتة الحجم — تُخصَّص مرة واحدة عند الإقلاع.
  • أي خرق = SLA breach → Monitor Only + تسجيل.

6.1 حقول الحدث الإلزامية

  • ts_rx — وقت الاستلام في Rust (بعد UDS receive).

  • رفض أي حدث بـ ts_src أقدم من 5 ثوان (stale source).

  • قياس الكمون: ts_src → ts_rx → ts_proc لكل حدث حرج.

7) Tiered Watchlist + TickReady Barrier

7.1 الطبقات

  • Tier A: 3–5 رموز — Tick-by-Tick دائمًا — مؤهلة للدخول.
  • Tier B: 5–15 رموز — Snapshots + Warm Buffer — مرشحة للترقية.
  • Tier C: بقية المرشحين — Slow Loop فقط — لا تداول مباشر.
قواعد التدرج:
  • لا دخول إلا من Tier A.
  • Tier C لا يرتقي مباشرة إلى A — يمر عبر B أولاً.
  • الترقية B→A: تتطلب TickReady + اجتياز §12.4 Guards ضد الترقية الزائفة.

7.2 IBKR Subscription Limits ⬡ v7.0

IBKR تُقيِّد عدد الـ concurrent live tick subscriptions (افتراضياً 50–100 رمز حسب نوع الحساب). SlowLoop يتتبع عدد الاشتراكات الفعلية ويمنع التجاوز:
  • عند الاقتراب من 80% من الحد: تنبيه + وقف ترقيات جديدة.
  • عند الوصول للحد: خفِّض Tier B (أسقط الأقل نشاطاً) لتحرير اشتراكات.
  • إذا رفض IBKR اشتراك Tick: لا تكرار عدواني — أعد الرمز Tier B فوراً.

7.3 TickReady Barrier (إغلاق Race Condition)

  • عند قرار الترقية B→A: اصدر أمر اشتراك Tick-by-Tick عبر Python.
  • لا تعتبر الرمز Tier A حتى وصول TickReady (أول Tick-by-Tick مؤكد).
  • قبل TickReady: الرمز يبقى Tier B حتى لو وصل Snapshot.
  • هذا يمنع FastLoop من معالجة رمز كـ Tier A قبل اكتمال اشتراكه.

7.4 Pacing Limits

  • SlowLoop يمتلك Budget محدود للطلبات في الدقيقة (قيمة من default.toml).
  • عند الاقتراب من الحد: خفِّض Tier B أولاً قبل رفض الترقيات.
  • Pacing violations تُسجَّل في MetricsTask.

9) Daily Context Filter (الخطوة 6) ★ تعريف كمي مُضاف

Gate إلزامي: فشل أي شرط = رفض فوري حتى لو كانت السيولة التاريخية ممتازة.

9.1 الشروط الكمية (كلها يجب أن تنطبق)

  • Volume_today ≥ 2× AvgVolume_20D. يُقاس كل 30 دقيقة في SlowLoop.
  • اتساق السعر والحجم: ارتفاع السعر يجب أن يُرافقه ارتفاع الحجم (لا ارتفاع على حجم هابط).
  • رفض Churning: High–Low < 1% دون تقدم للأعلى خلال 10 دقائق → Churning → رفض.

9.2 محفز اليوم — تعريف كمي قابل للبرمجة ⬡ v7.0

المحفز موجود إذا تحقق Volume_today > 2× AvgVolume_20D مع أي قرينة من الثلاث التالية:
  • Volume Surge: Volume_today > 3× AvgVolume_20D في أول ساعتين = محفز قوي (يكفي وحده).
  • News Event: حدث IBKR من نوع NewsEvent للرمز في الجلسة الحالية.
  • Sector Momentum: ETF الممثِّل للقطاع تحرك بـ ≥ 2% في الساعة السابقة.
Volume بين 2× و3× بدون أي من القرائن الثلاث → Context = UNDETERMINED → رفض. RejectReason = DAILY_CONTEXT.

يُميّز بين سهم "نشط اليوم" وسهم "تاريخه نشط". الثاني لا يُتداوَل.

10) MTF Context Analysis (الخطوة 7) ★ تعريفات كمية

Gate إلزامي: تعارض قوي مع اتجاه أعلى = Veto قاطع. (RejectReason: MTF_VETO)

10.1 الإطار الأسبوعي

  • Close الأسبوع الحالي > EMA(10 أسابيع) → اتجاه صاعد = مسموح.
  • Close < EMA(10w) بأكثر من 3% → Veto.

10.2 الإطار اليومي

  • مقاومة قريبة: مستوى شُكِّل بـ ≥ 2 لمسة خلال آخر 20 يوم، يبعد < 2% → Veto. اللمسة: High أو Low يومي يُلامس المستوى ± 0.5%.
  • Momentum slope: EMA(9d) < EMA(21d) بأكثر من 2% → Veto.

10.3 الإطار 4H

  • انحدار مستمر: 3 كابدلات 4H متتالية هابطة مع دخول = Veto.
  • Close آخر كابدل 4H > Close قبله = موجب أو محايد = مقبول.

10.4 الإطار 15 دقيقة

  • Pullback قبل الدخول: تراجع 1–3 شمعات من القمة مع انخفاض حجم ثم عودة بحجم متصاعد = مقبول.
  • Breakout مباشر بدون تصحيح: مقبول فقط إذا Volume_15m > 1.5× AvgVolume_15m.

10.5 ملاحظة تنفيذية

كل هذه المعايير تُحسب في SlowLoop وتُخزَّن كـ Boolean واحد (mtf_pass) في WatchlistSnapshot. FastLoop يقرأ الـ Boolean فقط — لا يُعيد الحساب أبداً.

10.6 تحفظ تطبيقي على أسهم السنتات ⬡ v7.0

أسهم السنتات قد تتحرك 200%+ في يومين. EMAs الأسبوعية قد لا تعكس بنية موثوقة. يُنصح بمراجعة نسبة Reject هذه Gate عبر Walk-Forward §26.3 لتقييم قدرتها التمييزية الفعلية. إذا تجاوزت نسبة رفض MTF 60% من إجمالي Rejects لأسبوعين متتاليين → مراجعة يدوية للمعايير.

12) Cold Start Solution

12.1 المشكلة

ترقية رمز من Tier B إلى A تُغيِّب نافذة Trades_prev20s اللازمة لحساب Acceleration. هدف الحل: منع قيم NaN/0 ومنع تفويت بداية الانفجار.

12.2 State Machine + Warm Buffer

  • States: COLD_START → WARM_ACTIVE → FULL_ACTIVE.
  • SlowLoop يُخزِّن: snap_trade_count_20s، snap_volume_20s، snap_spread_20s.
  • في WARM_ACTIVE: A_est = (last10s/10) ÷ max(snap_count/20, ε) حيث ε = 1e-6 لمنع القسمة على صفر.
  • وزن Acceleration في TapeScore أثناء WARM_ACTIVE = 50% من وزنه الكامل (wA_warm = 0.11).
  • بعد 20 ثانية من Tick حقيقي: → FULL_ACTIVE مع الأوزان الكاملة.

12.3 Surge Override (محكوم بشروط صارمة)

دخول في WARM_ACTIVE فقط إذا اجتمعت كلها: R ≥ 2.5 + LargePrint موجود + Spread stable + RunUp ≤ 3%. RunUp = ارتفاع السعر منذ بدء snap — يمنع مطاردة حركة منتهية.

12.4 Guards ضد الترقية الزائفة

  • لا ترقِ إلى Tier A إذا snap_trade_count_20s < حد أدنى (من الإعدادات).
  • لا ترقِ إذا snap_spread_20s فوق حد Universe Spread من §8.2.

13) Tape Engine + Ring Buffers ★ Absorption + BLS مُستعادان

13.1 مواصفات Ring Buffer

  • Ring Buffer زمني — نافذة W = 30 ثانية لكل رمز في Tier A.
  • حد أقصى: 2000 عنصر لكل رمز — تُخصَّص مرة واحدة عند الإقلاع.
  • عند الفيضان: إسقاط الأقدم (FIFO).
  • كل عنصر يحمل timestamp بدقة microsecond لحساب نوافذ 10s/20s.

13.2 مقاييس Tape — الحسابات الكمية

① Aggressive Ratio (R)

price ≥ ask  → Aggressive Buy  → V_buy  += volume
price ≤ bid  → Aggressive Sell → V_sell += volume
غير ذلك    → Neutral         → مُهمَل

R = (V_buy + 1) / (V_sell + 1)   # نافذة W = 30s
# R = 1.0 → توازن
# R > 1.5 → ضغط شراء
# R > 2.5 → ضغط قوي

② Acceleration (A)

A = (Trades_last10s / 10) / (Trades_prev20s / 20)
# A > 1.0 → تسارع
# A > 1.6 → نشاط حقيقي (FULL_ACTIVE)
# A > 1.9 → نشاط قوي (WARM_ACTIVE)

③ Large Print (LP)

LargePrint = TradeSizeUSD >= LargePrintThreshold

LargePrintCount_30s = count(LargePrint, 30s)
 True إذا صفقة حجمها ≥ max(P95_window, 4 × AvgTradeSize)
# P95 يُحسب على صفقات نافذة W الحالية

④ Absorption ★ مُستعاد

Absorption = True إذا:
  1. السعر ثابت قرب Ask (± max(0.5¢, 0.15% × Price)) لمدة ≥ 5s
  2. مع استمرار Aggressive Buys (V_buy > 0 كل ثانية)
  3. ثم ارتفاع السعر ≥ 1 tick خلال ثانيتين

ScoreAbsorption = 1.0 إذا تحققت جميع الشروط، 0.0 غير ذلك.
ملاحظة: الحد الأدنى ± max(0.5¢, 0.15% × Price) مُكيَّف مع السعر.
مثال: سعر 1$  → ± 1.5¢ | سعر 5$ → ± 7.5¢ | سعر 10$ → ± 1.5¢ (0.15%)

⑤ Bid Lifting Sequence (BLS) ★ مُستعاد

BLS = True إذا:
  ≥ 3 تحديثات NBBO متتالية ترفع الـ bid
  خلال نافذة 3–8 ثوان
  مع كل تحديث يرفع bid بـ ≥ 1¢

ScoreBidLift = 1.0 إذا تحققت، 0.0 غير ذلك.

13.3 دوال التطبيع (Normalization → 0 إلى 100)

ScoreR    = clamp((R - 1.0) / 2.0 * 100,  0, 100)
ScoreA    = clamp((A - 1.0) / 1.5 * 100,  0, 100)
ScoreLP   = min(LargePrintCount_30s * 25,   100)
ScoreSpread = clamp((SpreadLimit - SpreadPct) / SpreadLimit * 100, 0, 100)
              # أقصى إذا Spread = 0، صفر إذا عند الحد
ScoreAbs  = Absorption * 100.0
ScoreBLS  = BLS    * 100.0

13.4 Veto إضافي للـ Tape

انعكاس مفاجئ: R < 0.7 مع توسع سبريد خلال 5 ثوان → رفض دخول مباشر. RejectReason = TAPE_REVERSAL.

13.5 قواعد الأداء — O(1) مطلق

  • Ring Buffers ثابتة — تحديث counters بـ O(1) عبر head/tail pointers.
  • لا بحث خطي داخل FastLoop — لا loops فوق البيانات.
  • لا allocations بعد الإقلاع.

16) Position Sizing + Stops ★ MinStopDistance متعدد العوامل

16.1 StopDistance — الأولوية صريحة

Stop_Guards = max(2 * TickSize, 0.25 * (Ask - Bid))
Stop_ATR     = ATR(1m) × k_atr        # k_atr = 2.0 (ابتدائي، راجع §33)

StopDistance = max(Stop_Guards, Stop_ATR)

# حد أدنى إلزامي (MinStopDistance) — يمنع Position Sizes شاذة:
MinStopDistance = max(0.02$, 1.2% × Price, 0.8 × ATR_1m)
StopDistance    = max(StopDistance, MinStopDistance)

# ملاحظة: MinStopDistance مُكيَّف مع التقلب (ATR) والسعر معاً.

16.2 حساب الحجم

RiskPerTradeUSD    = 25$  (ثابت قابل للضبط، راجع §33)
SharesRaw          = RiskPerTradeUSD / StopDistance
SharesCapLiquidity = (ADDV_usd × 0.1%) / Price
BudgetCap          = SettledCash × 20%
MaxPositionValue   = AccountEquity × 15%   # Hard Reject إذا تجاوز

Shares = min(SharesRaw, SharesCapLiquidity, BudgetCap)
# التحقق الأخير: Shares × Price ≤ MaxPositionValue

16.3 تعديل الحجم بعد الهدف اليومي

  • بعد تحقيق الهدف اليومي: Shares *= 0.50.
  • بعد تحقيق 10% ربح يومي إجمالي: Shares *= 0.25.

17) Exposure + Correlation (الخطوة 12)

  • MaxOpenPositions = 1 (افتراضي).
  • يُسمح بـ 2 فقط إذا: Regime = Normal مستقر + Correlation منخفض + لا تركز قطاعي.
  • إذا نفس القطاع (SIC/GICS): MaxOpenPositions = 1 بصرف النظر.

17.1 Correlation Guard — التنفيذ الكامل

  • يُحسب Pearson(20d daily returns) في SlowLoop — خارج FastLoop.
  • عند وجود مركز مفتوح: SlowLoop يحسب Correlation بين الرمز المفتوح وكل مرشح Tier A ويخزن في watchlist_snapshot.corr_to_open.
  • إعادة الحساب: مرة واحدة عند تغيّر الرمز المفتوح، ثم إعادة استخدام حتى التغيير التالي.
  • FastLoop يقرأ قيمة جاهزة فقط: إذا corr_to_open ≥ 0.40 → رفض المركز الثاني.

Threshold = 0.40 اختير كحد محافظ مناسب لـ Penny Stocks ذات تباين عالٍ. (راجع §32 Rationale)

18) Fees + Slippage + ExpectedNet (الخطوة 11)

توضيح Gross — بدون لبس:
  • TargetPriceMove = EntryPrice × 0.10
  • TargetProfit_10pct = TargetPriceMove × Shares
  • Gross = min(50.0$, TargetProfit_10pct)
  • ExpectedNet يُحسب على الكمية الكاملة. البيع الجزئي 50% يخص إدارة الصفقة فقط (§19.4).

18.1 معادلة الرسوم

Commission = 0.005$ × shares                      # IBKR per-share
SEC_fee    = 0.0000278 × (sell_price × shares)    # معدل 2025 — يتغير سنوياً
TAF        = 0.000166 × shares                    # مع سقف تنظيمي
Routing    = 0$ (IBKR Smart) أو من الإعدادات

TotalFees  = Commission + SEC_fee + TAF + Routing

معدل SEC_fee يتغير سنوياً — تحقق من default.toml كل يناير.

18.2 نموذج الانزلاق

ImpactSlippage   = β × (Shares / AvgDepthTop3)
SlippagePerShare = max(Spread/2, α × Vol_1m × Price, ImpactSlippage)
ExpectedSlippage = SlippagePerShare × shares

# قيم ابتدائية: α = 0.5، β = 0.3
# تُعاير عبر Live Micro Testing §26.4

18.3 شرط القبول — Veto قاطع

ExpectedNet = Gross - TotalFees - ExpectedSlippage
ExpectedNet ≤ 0 → رفض فوري. RejectReason = NET_NEGATIVE.

19) Execution Engine + OMS

19.1 أنواع الأوامر

  • Bracket Orders + OCO (One-Cancels-Other).
  • Server-side Stops — يُرسَل مع أمر الدخول قبل إغلاق OMS.
  • Trailing Stop ديناميكي بعد تحقيق الهدف الأول.

19.2 Partial Fills

  • تتبع filled_qty وavg_fill_price في الوقت الحقيقي.
  • لا إعادة إرسال دون idempotency key فريد (UUID أو timestamp + symbol_id).
  • Repricing: حد أقصى 3 تعديلات، فاصل 250ms. بعد الحد: Cancel + Veto.

19.3 Fill Confirmation Timeout

  • Timeout = 2 ثانية بعد إرسال الأمر.
  • عند التجاوز: Query order status فوراً.
  • إذا لم يتأكد خلال 5 ثوان → INVESTIGATE + منع أوامر جديدة للرمز + تنبيه.

19.4 إدارة الصفقة بعد الدخول

  • عند تحقيق 50$ Gross أو 10% تحرك سعري: بيع 50% (Partial Exit — Bid-based Realized PnL).
  • الباقي: رفع Stop إلى Breakeven + Trailing Stop ديناميكي.
  • عند Tape Reversal (R < 0.7) أو Spread Widening مفاجئ: تشديد Stop أو خروج فوري.

20) Risk Engine + Kill Switch

20.1 Loss Ladder

  • خسارة 1 في اليوم: Pause 15 دقيقة.
  • خسارة 2 في اليوم: Pause 1 ساعة.
  • خسارة 3 أو PnLDay ≤ −100$: Monitor Only حتى بداية الجلسة التالية.

20.2 أولويات Kill Switch (مرتبة تنازلياً)

  1. MaxDailyLossUSD = −100$
  2. Data/API failure (DataQuality = degraded > 30s)
  3. Loss Ladder (3 خسارات)
  4. SLA breach مستمر (> 25ms لمدة > 5s)
  5. Correlation breach (مركزان متشابهان)

20.3 LULD / Halt Policy ✦ سياسة موحَّدة نهائية

السياسة Symbol-specific — Halt على رمز لا يوقف تداول باقي الرموز.
  • إذا يوجد مركز مفتوح: ننتظر حتى 5 دقائق. إذا استمر الـ Halt > 5 دقائق → EMERGENCY_EXIT فور الاستئناف (Marketable Limit 
    ExitPrice = max(
    0.01,
    Bid - 0.25 * ATR_1m,
    Bid - 2 * TickSize
)
    — لا تعتمد على Stop كسعر مضمون).
  • دخول جديد بعد أي Halt/LULD: تعطيل الرمز فوراً لبقية الجلسة. (DISABLE_SYMBOL_REMAINDER_SESSION)
  • Gap بعد الاستئناف: إذا ResumePrice < PreHaltMark × 0.92 → تأكيد التعطيل + تسجيل GAP_AFTER_HALT_DISABLE.

20.4 Gap Risk Protection

  • Open Gap Filter: إذا OpenPrice < PrevClose × 0.92 → Monitor Only للرمز لمدة 15 دقيقة. تسجيل GAP_OPEN_FILTER. مراكز قائمة تُدار بقواعد الخروج كما هي.
  • Gap filters تطبق على مستوى الرمز فقط — لا تأثير على باقي الرموز.

20.5 تفاعل قواعد الخسارة مع المراكز المتزامنة

  1. MaxDailyLoss يُحتسب كمجموع صافٍ لجميع المراكز — يسبق كل شيء.
  2. كل خسارة رمز مستقلة تُحتسب في Loss Ladder بشكل مستقل.
  3. مجموع الخسائر اليومية يُقاس مقابل MaxDailyLossUSD.

21) T+1 Settlement + PDT Rule ⬡ PDT مُضاف في v7.0

21.1 قواعد التسوية

  • التداول فقط من Settled Cash.
  • منع Good Faith Violations تلقائياً — لا شراء بأموال غير مستقرة.
  • تتبع Settled/Unsettled Cash بعد كل صفقة.
  • أي منع بسبب Settlement يُسجَّل في DecisionLog مع السبب.

21.2 Pattern Day Trader (PDT) Rule ⬡ v7.0

Cash Account يخضع لقاعدة الـ Settlement وليس PDT بالمعنى التقني (PDT يُطبَّق على Margin Accounts). لكن يجب مراقبة:
  • عدد الصفقات اليومية لا يتجاوز ما يُتيحه Settled Cash.
  • إذا تجاوز الحساب 25,000$ وتحوَّل إلى Margin: يجب مراجعة قواعد PDT فوراً ومنع التداول حتى التحقق.
  • النظام يُسجِّل عدد الصفقات اليومية ويُنبِّه عند الاقتراب من حدود بيئة الحساب.

  2. Verify Stop/OCO موجود لكل مركز مفتوح.

    0.01,
    Bid - 0.25 * ATR_1m,

    أو Market Stop) قبل أي شيء آخر.
  4. Rebuild State Machine من IBKR truth — لا من الذاكرة المحلية.
  5. No new orders حتى اكتمال خطوات 1–4.

26.1 Test Harness

  • Replay Engine: تشغيل بيانات تاريخية بترتيب حرفي (tick-by-tick).
  • Deterministic Clock: الزمن من البيانات لا من النظام.
  • Golden Decision Files: قرارات مرجعية لاختبار Regression بعد كل تغيير.

26.2 Stress Scenarios (إلزامية)

  • Burst ticks (1000 tick/s لمدة 30s).
  • Spread widening مفاجئ (×5 خلال ثانية).
  • Stale quotes (NBBO ثابت 10s مع trades).
  • Partial fills متعددة على نفس الأمر.
  • Reconnect أثناء صفقة مفتوحة.
  • LULD Halt ثم استئناف بعد 4 دقائق و بعد 6 دقائق.
  • Regime change Normal→Caution→Risk-Off أثناء صفقة.
  • Gap down >8% عند الافتتاح.

26.3 Walk-Forward Methodology

  • تقسيم: 70% In-Sample / 30% Out-of-Sample.
  • نافذة: 20 يوم تداول، خطوة 5 أيام (rolling).
  • منع Lookahead Bias: ATR وVolume Averages من In-Sample فقط.
  • مقاييس: MaxDrawdown، Sharpe Ratio، Win Rate per Regime، Avg Slippage Ratio.
  • معايرة: TapeScore thresholds وأوزان تُعدَّل فقط بعد Walk-Forward + توثيق مكتوب.

26.4 Live Micro Testing

  • أحجام 10–50 سهم لمعايرة α، β والرسوم الفعلية.
  • مقارنة Slippage متوقع vs فعلي لكل صفقة.
  • تحديث default.toml بعد كل دورة معايرة.

32) Engineering Rationale ⬡ جديد كلياً في v7.0

يُوثِّق هذا القسم سبب القرارات الهندسية الرئيسية لمنع الرجوع عنها بدون دليل.

32.1 لماذا Cold Start لم يُحذف؟

رغم تعقيده، يحمي النظام من الدخول على بيانات ناقصة عند ترقية Tier. ε = 1e-6 يمنع القسمة على صفر. الحل محافظ ومُعقَّد بشكل متعمد.

32.2 لماذا Correlation Threshold = 0.40؟

ليس عشوائياً — متسق مع معايير إدارة المخاطر في Prop Desks صغيرة. Penny Stocks ذات تباين عالٍ → threshold منخفض يُولِّد false rejects كثيرة. 0.40 توازن بين حماية التركز وتوفير فرص كافية.

32.3 لماذا MinStopDistance متعدد العوامل؟

الجمع بين نسبة السعر (1.2%) وATR (0.8×) و0.02$ الثابت: يمنع وقفاً ضيقاً جداً يؤدي لـ Position Size ضخمة في أسهم منخفضة التقلب اللحظي. RiskPerTrade يبقى ثابتاً (25$) عبر حجم المركز عند تطبيق المعادلة.

32.4 لماذا Internal SLA = 10ms وليس 100ms؟

10ms هدف هندسي داخلي (FastLoop processing) — يُقاس من استلام الحدث حتى اتخاذ القرار.
25ms حد Hard Fail داخلي — إذا تجاوز لأكثر من 5 ثوان → Monitor Only.
100ms Hard Max End-to-End (Tick → Order sent، شامل Python Bridge + UDS) — بعده Monitor Only.
الثلاثة ليست متعارضة — كل منها يقيس مستوى مختلفاً من المسار. (راجع §22)

32.5 لماذا Python Bridge وليس كل شيء في Rust؟

ib_insync هو المكتبة الأكثر نضجاً للاتصال بـ IBKR TWS. الكتابة في Rust خام تتطلب reverse engineering لبروتوكول IBKR غير موثَّق كاملاً. التكلفة: UDS latency (~0.1ms على نفس الجهاز) — مقبولة جداً مقارنة بالبديل.

32.6 لماذا TapeScore أولاً وليس ExpectedNet؟

ExpectedNet يعتمد على Shares — والـ Shares تعتمد على StopDistance. TapeScore أسرع حسابياً (O(1) من Ring Buffers جاهزة). رفض على TapeScore أرخص من حساب كامل لـ ExpectedNet. الترتيب يُحسِّن الأداء دون المساس بسلامة القرار.
